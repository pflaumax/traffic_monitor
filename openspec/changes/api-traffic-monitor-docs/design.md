## Context

The API Traffic Monitor is a fully implemented HTTP reverse proxy (commit `a32b980`, branch: `development`) that forwards requests to httpbin.org, emits traffic events to Kafka, and writes aggregated metrics to Redis. The core proxy is complete, but the project has no formal specifications, no structured task backlog, and no documented architectural decisions. The remaining roadmap (rate limiter, Kafka consumer, dashboard, expanded tests) cannot be implemented consistently without a shared specification contract.

This design covers the documentation generation effort — creating 7 spec documents that capture the current system and define the remaining work.

## Goals / Non-Goals

**Goals:**
- Produce a complete, accurate specification suite derived from the master project summary (`docs/API-Traffic-Monitor-Master-Project-Summary.md`)
- Document all implemented behavior as ground truth (no invention)
- Formally capture all open architectural questions so they can be resolved before roadmap work begins
- Create a structured task backlog that distinguishes DONE, TODO, BUG, and DECISION items
- Design the dashboard UI with phased implementation plan
- Provide a single environment/config reference for all contributors

**Non-Goals:**
- Implementing any new features (rate limiter, consumer, dashboard code)
- Changing any existing production code in `proxy/`, `shared/`, or `tests/`
- Resolving open architectural questions (document them, not decide them)
- Generating test cases or CI configuration

## Decisions

### Decision 1: Source of Truth
**Choice:** `docs/API-Traffic-Monitor-Master-Project-Summary.md` is the sole source of truth for all generated specs.

**Rationale:** This document was generated by auditing the actual codebase at commit `a32b980` and cross-referencing two earlier summaries. It explicitly notes where code and docs conflicted and trusted the code. Using it as the single source prevents contradictions across spec files.

**Alternative considered:** Reading source files directly (`proxy/main.py`, `proxy/redis_client.py`, etc.) — rejected because the master summary already did this work and is more complete than any single file.

### Decision 2: Spec File Structure
**Choice:** One spec file per capability (`specs/<capability>/spec.md`), matching the 7 capabilities listed in the proposal.

**Rationale:** Keeps each spec focused and independently navigable. The capabilities map cleanly to the 7 documents requested in `docs/openspec_prompt.md`.

**Alternative considered:** Single monolithic spec file — rejected because it would be unwieldy and harder to reference from tasks.

### Decision 3: Task Status Taxonomy
**Choice:** Four task statuses: DONE (completed commits), TODO (roadmap items), BUG (known issues), DECISION (open architecture questions).

**Rationale:** Directly maps to the four categories in the master project summary (commit history, roadmap, known issues, open questions). Makes the backlog immediately actionable.

### Decision 4: Dashboard Tech Stack Recommendation
**Choice:** Recommend Option A (HTMX + FastAPI) as primary, Option B (static HTML) as fast demo fallback.

**Rationale:** HTMX fits the existing Python/FastAPI stack with no new language or build tooling. Option B (static HTML + Chart.js) is documented as a 2-hour demo path. Option C (Grafana) is noted as a future Phase 3 option.

### Decision 5: Open Questions — Document, Don't Resolve
**Choice:** All 6 open architectural questions from the master summary are preserved verbatim in the architecture spec as `[OPEN QUESTION]` items.

**Rationale:** These questions (direct Redis writes vs. consumer, TTL on stats keys, raw counters vs. rates, etc.) require team discussion and potentially code changes. Resolving them unilaterally in a documentation pass would be premature.

## Risks / Trade-offs

- **Risk: Spec drift from code** → The master summary is dated 2026-05-02 at commit `a32b980`. If the codebase advances before specs are used, they may become stale. Mitigation: specs reference specific commit hashes; reviewers should re-verify against code before implementing roadmap tasks.

- **Risk: Over-specification of unbuilt features** → Dashboard and consumer specs describe planned behavior that doesn't exist yet. Mitigation: all planned items are clearly marked 🔲 TODO; specs describe intent, not current behavior.

- **Risk: Open questions block roadmap tasks** → Several TODO tasks (Kafka consumer, Redis TTL) depend on unresolved architectural decisions. Mitigation: DECISION tasks are listed first in the backlog with explicit blocking relationships noted.
